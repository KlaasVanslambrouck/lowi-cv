## 4. Zelf te beoordelen punten (niet gefixt)

1. **Admin-skip handmatig natesten.** Het "ingelogde admin genereert geen
   analytics"-mechanisme is code-correct maar kon zonder jouw admin-wachtwoord
   niet end-to-end worden getest. Doe dit één keer zelf: log in op `/beheer`,
   browse de publieke site in dezelfde browser, en check in Supabase dat er
   geen rijen bijkomen (de API antwoordt dan `{"ok":true,"skipped":true}` —
   zichtbaar in het Network-tabblad). Let op: het skip-pad werkt op de
   Supabase-cookies, dus alleen in de browser waarin je bent ingelogd.

2. **De login-rate-limit is een voorportaal, geen harde grens.** De teller zit
   in een apart endpoint (`/api/auth/login-attempt`) dat het loginformulier
   vrijwillig aanroept vóór `signInWithPassword`; wie Supabase rechtstreeks
   met de (publieke) anon key benadert, passeert hem. Supabase heeft zelf
   auth-rate-limits als vangnet, en de in-memory teller reset bij serverless
   cold starts (beide beperkingen staan eerlijk gedocumenteerd in code en
   SECURITY.md). Wil je dit hard maken, dan moet de login server-side lopen
   (route handler die zelf `signInWithPassword` uitvoert) — architectuurkeuze,
   niet ongevraagd doorgevoerd.

3. **De globale track-noodrem (240 events/min) is een gedeelde cap.** Eén
   spammer met steeds nieuwe sessie-id's kan hem vullen en daarmee analytics
   van echte bezoekers tijdelijk dempen. Bewuste afweging (geen IP-opslag =
   geen per-client-identiteit) met een TODO in de code; voor dit
   traffic-niveau verdedigbaar.

4. **`THREE.Clock`-deprecation-warning op WebGL-desktops** komt uit
   `@react-three/fiber` (intern `new THREE.Clock()`), niet uit projectcode.
   Verdwijnt vanzelf bij een toekomstige fiber-update; zelf niets aan te doen
   behalve de dependency t.z.t. bumpen.

5. **RLS is niet vanuit de repo verifieerbaar.** Er zijn geen migrations;
   [docs/database-security.md](docs/database-security.md) beschrijft het
   verwachte contract en geeft kant-en-klare verificatiequeries. Draai die
   één keer tegen je productie-Supabase om te bevestigen dat RLS aanstaat en
   er geen anon-policies op `portfolio_analytics` staan.

6. **JarvisPresence (en dus de hele proactieve laag) is desktop-only** —
   op <768px is het paneel verborgen (bestaande keuze uit de mobiele audit).
   Mobiele bezoekers zien de reactieve laag dus nooit; dat is consistent met
   "subtiel", maar goed om bewust van te zijn.

7. **De admin-user-id staat hardcoded in [lib/auth/admin.ts:7](lib/auth/admin.ts#L7)**,
   met expliciete motivering (geen secret, beter reviewbaar dan een env-var).
   Verdedigbaar voor een single-admin-app; verhuizen naar een env-var kan als
   je ooit van Supabase-project wisselt.

8. **De beheer-loginpagina is Nederlandstalig-only** (labels "E-mail",
   "Wachtwoord", "Aanmelden") terwijl de rest van de site tweetalig is. Voor
   een pagina die alleen jij ziet lijkt dat bewust; strikt genomen wijkt het
   af van het bilingual-patroon.

## 5. Afwijkingen van de oorspronkelijke opdrachten (bewust, geen fouten)

- **`proxy.ts` i.p.v. `middleware.ts`** — dit is simpelweg de nieuwe naam van
  hetzelfde mechanisme in Next.js 16; de build bevestigt dat hij actief is.
  Functioneel exact wat gevraagd was.
- **Jarvis-drempel op precies 15s** — de spec zei "~15-20s"; 15 valt binnen de
  marge (ondergrens).
- **Login via client-side `signInWithPassword`** met een server-side
  pre-flight-gate, i.p.v. een volledig server-side loginflow. De échte
  beveiligingsgrens (dashboard-toegang) is wél volledig server-side; zie §4.2.
- **Extra's bovenop de prompts**: security headers + optionele HSTS,
  error boundaries, ARCHITECTURE/SECURITY/database-security-documentatie,
  method-guards (405) en content-type/body-size-checks op de track-API.
  Allemaal in lijn met de geest van de opdrachten.
